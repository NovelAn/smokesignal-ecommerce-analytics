## 🤖 AI分析功能完善 (2025-01-20)

### 问题发现

用户反馈了三个关键问题：
1. **AI分析数据错误** - mylifemyrule客户历史消费6500元，但AI总结显示"历史消费总额为0"
2. **AI洞察质量不足** - AI仅重复已有数据（城市、消费金额），未提供深度洞察
3. **前端模块数据缺失** - Purchase History和Communication标签页显示无数据

### 根本原因分析

**问题1：字段名不匹配**
- API返回`historical_net_sales`，但AI提示词期望`historical_ltv`
- 影响：AI无法获取正确的消费数据
- 涉及文件：`backend/ai/zhipu_client.py:85`，`backend/api/target_routes.py:355`

**问题2：AI提示词未明确分析要求**
- 原提示词仅要求"总结客户信息"，未强调深度分析
- AI倾向重复已有数据，而非挖掘行为模式和心理动机
- 涉及文件：`backend/ai/zhipu_client.py:67-113`

**问题3：前端数据流断裂**

- `groupedMessages`使用错误数据源（`currentSession.messages`而非`enrichedProfile.chat_history`）

## 💬 聊天指标增强 (2025-01-21)

### 需求背景

在之前的AI分析功能完善中，Communication模块已经能够正确显示聊天记录，但是缺少深入的沟通行为分析指标。为了更好地评估客户活跃度和沟通频次，需要添加以下关键指标：

1. **首次聊天时间** - 了解客户建立联系的时间点
2. **近30天沟通频次** - 评估近期活跃度（去重日期天数）
3. **平均沟通间隔** - 评估客户沟通习惯

### 实施方案

**数据库层面（MySQL迁移）**

创建迁移脚本 `backend/database/sql/add_chat_metrics_to_precomputed.sql`：

```sql
-- 添加3个新字段
ALTER TABLE target_buyers_precomputed
ADD COLUMN first_chat_date DATETIME COMMENT '首次聊天时间' AFTER last_chat_date,
ADD COLUMN l30d_chat_frequency_days INT COMMENT '近30天沟通频次(去重日期天数)' AFTER first_chat_date,
ADD COLUMN avg_chat_interval_days DECIMAL(10, 2) COMMENT '平均沟通间隔天数' AFTER l3m_chat_frequency_days;

-- 更新现有数据（404条记录）
UPDATE target_buyers_precomputed tb
SET
    first_chat_date = (SELECT MIN(msg_time) FROM chat_history WHERE user_nick = tb.buyer_nick),
    l30d_chat_frequency_days = (
        SELECT COUNT(DISTINCT DATE(msg_time))
        FROM chat_history
        WHERE user_nick = tb.buyer_nick
          AND msg_time >= DATE_SUB(NOW(), INTERVAL 30 DAY)
    ),
    avg_chat_interval_days = (
        SELECT CASE
            WHEN COUNT(DISTINCT DATE(msg_time)) <= 1 THEN 0
            ELSE DATEDIFF(MAX(DATE(msg_time)), MIN(DATE(msg_time))) / (COUNT(DISTINCT DATE(msg_time)) - 1)
        END
        FROM chat_history
        WHERE user_nick = tb.buyer_nick
    );
```

**后端API更新**

更新查询文件 `backend/database/sql/target_buyers/get_target_buyer_by_nick.sql`，添加新字段到SELECT语句。

### 遇到的问题和解决

**问题1：前端显示0值**
- **现象**：所有聊天指标显示为0
- **原因**：`enrichedProfile`未包含API返回的新字段
- **解决**：更新enrichedProfile的useMemo依赖，添加所有新字段

**问题2：TypeError - .toFixed is not a function**
- **现象**：浏览器控制台报错
- **原因**：MySQL DECIMAL类型在JSON中返回为字符串"1.00"，无法直接调用.toFixed()
- **解决**：使用Number()包装：`Number(value || 0).toFixed(1)`

**问题3：discount_ratio显示异常**
- **现象**：折扣率计算错误
- **原因**：
  1. Operator precedence: `|| 0 * 100` = `|| (0 * 100)` = `|| 0`
  2. discount_ratio也是字符串类型
- **解决**：`(Number(value || 0) * 100).toFixed(0)`

### 技术收获

1. **MySQL类型映射**：DECIMAL → JSON字符串，需要显式转换
2. **React数据流**：确保useMemo依赖包含所有数据源
3. **Operator precedence**：逻辑或运算符优先级低于算术运算
4. **端到端测试**：Playwright MCP能快速发现类型错误

### 根本原因分析

**问题1：前端字段映射错误**
- `historical_ltv` 被映射为 `historical_net_sales` 而非 `historical_gmv`
- 影响：LTV显示为净销售额而非总交易额
- 涉及文件：`src/App.tsx:569`, `src/App.tsx:728`

**问题2：后端SQL缺少字段**
- `get_target_buyer_by_nick.sql` 查询未包含 `l6m_refund_rate` 和 `l1y_refund_rate`
- 影响：前端无法获取退款率数据
- 涉及文件：`backend/database/sql/target_buyers/get_target_buyer_by_nick.sql`

**问题3：订单显示逻辑问题**
- 前端将同一订单号下的多个商品合并成一行显示
- 影响：无法看清每个商品的具体信息
- 涉及文件：`src/App.tsx:628-683`
