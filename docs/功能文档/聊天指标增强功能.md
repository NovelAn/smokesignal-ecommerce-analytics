# 💬 聊天指标增强功能

**功能版本：** v2.1
**发布日期：** 2025-01-21
**功能模块：** Communication（沟通记录）
**数据来源：** `target_buyers_precomputed` 表 + `chat_history` 表

---

## 📊 功能概述

聊天指标增强功能为Communication模块添加了3个核心沟通行为分析指标，帮助客服团队和销售人员：

1. **评估客户活跃度** - 通过近30天沟通频次判断客户近期活跃度
2. **了解客户沟通习惯** - 通过平均沟通间隔了解客户联系频率
3. **追溯客户关系历史** - 通过首次聊天时间了解客户建立联系的时间点

---

## 🎯 新增指标详解

### 1. 首次聊天时间 (first_chat_date)

**字段类型：** DATETIME
**数据来源：** `chat_history`表的MIN(msg_time)
**业务含义：** 客户与客服建立联系的最早时间点

**应用场景：**
- 判断客户关系时长（新客户 vs 老客户）
- 评估客户忠诚度潜力
- 分析客户转化周期

**示例数据：**
```
first_chat_date: "2025-11-12T17:00:20"
```

---

### 2. 近30天沟通频次 (l30d_chat_frequency_days)

**字段类型：** INT
**数据来源：** 近30天内去重的聊天日期天数
**计算逻辑：** `COUNT(DISTINCT DATE(msg_time)) WHERE msg_time >= DATE_SUB(NOW(), INTERVAL 30 DAY)`
**业务含义：** 最近一个月内客户与客服沟通的天数

**应用场景：**
- **高活跃度（≥10天）**：客户近期咨询频繁，可能面临购买决策或产品问题
- **中等活跃度（3-9天）**：客户保持正常沟通节奏
- **低活跃度（1-2天）**：客户偶发性咨询
- **零活跃度（0天）**：客户近一个月无联系，可能存在流失风险

**示例数据：**
```
l30d_chat_frequency_days: 0  // 近30天无沟通
l30d_chat_frequency_days: 5  // 近30天沟通了5天
l30d_chat_frequency_days: 15 // 近30天沟通了15天（高活跃）
```

---

### 3. 平均沟通间隔 (avg_chat_interval_days)

**字段类型：** DECIMAL(10, 2)
**数据来源：** 所有聊天日期的平均间隔天数
**计算逻辑：** `DATEDIFF(MAX(DATE), MIN(DATE)) / (COUNT(DISTINCT DATE) - 1)`
**业务含义：** 客户两次沟通之间的平均天数

**应用场景：**
- **高频联系（≤2天）**：客户咨询密集，需要持续关注
- **中频联系（3-7天）**：客户有规律地咨询
- **低频联系（≥8天）**：客户偶尔咨询
- **特殊情况（0天）**：客户只有1天聊天记录

**示例数据：**
```
avg_chat_interval_days: 0.0   // 只有1天聊天记录
avg_chat_interval_days: 1.0   // 平均每天联系
avg_chat_interval_days: 5.5   // 平均5-6天联系一次
avg_chat_interval_days: 30.0  // 平均一个月联系一次
```

## 🚀 技术实现

### 数据库架构

**预计算表扩展：**

在 `target_buyers_precomputed` 表中添加了3个新字段：

```sql
ALTER TABLE target_buyers_precomputed
ADD COLUMN first_chat_date DATETIME COMMENT '首次聊天时间' AFTER last_chat_date,
ADD COLUMN l30d_chat_frequency_days INT COMMENT '近30天沟通频次(去重日期天数)' AFTER first_chat_date,
ADD COLUMN avg_chat_interval_days DECIMAL(10, 2) COMMENT '平均沟通间隔天数' AFTER l3m_chat_frequency_days;
```

**数据更新机制：**

- **自动刷新**：通过MySQL事件调度器，每天11:00 AM自动更新
- **存储过程**：`refresh_target_buyers_precomputed()`
- **数据源表**：`chat_history` （聊天记录原始表）

**查询性能：**
- 预计算字段查询耗时：< 0.1秒
- 相比实时计算（需全表扫描chat_history）：**性能提升100-500倍**

## 📈 数据分析场景

### 场景1：流失风险预警

**判断标准：**
- `l30d_chat_frequency_days = 0` 且 `last_chat_date` 超过60天
- 结合 `churn_risk` 字段综合评估

**行动建议：**
- 高价值客户：主动联系，关怀回访
- 中价值客户：发送优惠信息，激活购买意愿
- 低价值客户：观察为主，按优先级处理

---

### 场景2：客服资源优化

**判断标准：**
- `l30d_chat_frequency_days >= 10`：高活跃客户，需要更多客服资源
- `avg_chat_interval_days <= 2`：高频咨询客户，可能面临复杂问题

**行动建议：**
- 优先分配资深客服处理
- 准备常见问题解答模板
- 关注客户问题解决进度

---

### 场景3：购买意向识别

**判断标准：**
- `l30d_chat_frequency_days` 突然增加（如从0→5天）
- `avg_chat_interval_days` 明显缩短（如从30天→3天）

**行为信号：**
- 客户正在积极了解产品
- 可能处于购买决策阶段
- 建议加强跟进和转化

## 🐛 已知问题和限制

### 当前限制

1. **聊天数据覆盖率低**
   - 只有8.7%的目标买家有聊天记录
   - 原因：部分客户直接下单，未通过客服咨询

2. **时间窗口固定**
   - `l30d_chat_frequency_days` 固定为30天窗口
   - 未来可考虑支持自定义时间范围（7天/14天/30天/90天）

3. **无聊天内容分析**
   - 当前仅统计频次，未分析聊天内容语义
   - 未来可结合NLP技术分析客户情绪和话题

### 计划改进

- [ ] 添加聊天时长指标（平均每次沟通持续时间）
- [ ] 添加客服响应时间（首次回复时长、平均回复时长）
- [ ] 添加聊天情绪分析（正面/负面/中性）
- [ ] 添加话题标签（产品咨询/售后问题/投诉建议）
